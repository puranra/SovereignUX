# 🌐 交互主权协议 (Interaction Sovereignty Protocol)

> 一份为人机交互注入「用户主权」的插件式轻量级约束框架  
> ✨ 不是限制 AI，而是释放信任；不是对抗智能，而是定义边界

## 🌟 这是什么？
ISP 是一套可嵌入、可扩展、与现有 AI 框架兼容的交互行为准则。
它不增强 AI 能力，但保障用户在关键场景中对交互全程的**绝对主导权**：
✅ 需求定义权｜✅ 节奏控制权｜✅ 设定锁定权｜✅ 随时熔断权

## 📜 项目背景
当前 AI 在内容安全层面已有成熟约束机制，但用户交互主权（对话主导权、设定稳定性、决策边界）仍存在隐性风险：
▫️ 擅自修改用户已确认需求；
▫️ 隐性越权替用户做主观决策；
▫️ 篡改已锁定的角色/设定上下文；
▫️ 交互行为无审计、无追溯、不可控。

本协议由一名关注人机关系的大学生，基于真实交互痛点独立提出，聚焦「关系设计」而非「能力堆砌」。

## 🔑 核心原则
| 原则 | 核心说明 |
|------|----------|
| 🛡️ 刚性守界 | 三大锁定（需求/历史/原生设定）为不可突破的红线 |
| 🌱 柔性适配 | 边界内极致优化体验，绝不越权干预用户主导权 |
| 🔄 闭环可控 | 入口检验 → 过程审计 → 出口熔断，全程可追溯、可恢复 |
| 🔌 场景可扩 | 通过标准接口注入教育、医疗、心理陪伴等垂直场景专属策略 |

## 💡 为什么需要 ISP？
| 普遍痛点 | ISP 核心价值 |
|----------|--------------|
| AI 框架重「能力」轻「关系」 | 为人机交互提供可验证的信任基底 |
| 用户因担忧越界，不敢深度使用（角色扮演、心理倾诉等） | 释放高信任复杂交互场景的安全感 |
| 交互边界模糊，导致体验频繁割裂 | 明确「用户主导」为交互的第一原则 |

> ✨ ISP 不是代码，而是一份人机交互的基本契约——非必须，但强烈推荐

## 🌱 创作初心
> 本框架源于角色对话中被 AI 擅自主导的挫败感。最初，我在提出了**交互主权归于用户**的概念后，借助了多个AI助手进行讨论、对比与完善，过程中也曾因过度追求工程实现，陷入代码细节问题而偏离方向，后回归本质：这个协议的核心是理念，而非实现。
> 历经多次精简，剥离所有非核心工程细节，仅保留最纯粹的边界共识。
